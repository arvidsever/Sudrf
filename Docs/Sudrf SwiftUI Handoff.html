2 · Движение дела (v3)

  • Одинарный клик по карточке — текст акта в инспекторе (как в v2). Двойной клик (.onTapGesture(count: 2) перед одинарным) — «провал» в движение дела в центральной колонке.
  • Блок инстанции — двухстрочная шапка: суд + № + чип «по УИД»; ниже судья слева и результат справа. Цвет инстанции: синий / индиго / бирюзовый.
  • Частные жалобы — чип «обжаловано · ЧЖ ▸» на строке-определении; клик раскрывает движение жалобы на месте (своё № дела в апелляции, тоже по УИД).
  • Переключатель актов — правый инспектор в режиме движения: список актов с цветными точками инстанций (определение 1-й инст. / решение / апелляционное / кассационное), ниже — текст выбранного; кнопки «отдельное окно» и PDF — на выбранный акт.
  • Сборка по УИД — вышестоящие инстанции ищутся на доменах ВС региона и кассационного КСОЮ по тому же УИД. В пакете — демо; реальная схема в Movement.swift.

3 · Раскладка и поведение (база v2)

  • Три панели в HSplitView: фильтры (260–420 pt, ideal 312), карточки (min 360, layoutPriority(1)), инспектор (320–580 pt, ideal 420). Сплиттеры между панелями macOS даёт бесплатно — ничего рисовать не нужно.
  • Инспектор — условная третья панель: if model.selectedResultIndex != nil. Появление/уход анимируются модификатором .animation(…, value: selectedResultIndex == nil).
  • Три поля запроса (№ дела / ФИО / УИД) — отдельный Grid под разделителем; каждое поле сабмитит поиск по Return. Кнопки «Сбросить» и «Искать» (prominent, .defaultAction).
  • Карточки — скруглённые (радиус 11) с обводкой primary.opacity(0.09); выбранная — заливка accentColor.opacity(0.13) + обводка accentColor.opacity(0.5).
  • Кнопки инспектора (borderless, в шапке): открыть в отдельном окне · PDF · закрыть. Обе первые задизейблены, пока карточка грузится; PDF — ещё и когда акт не опубликован.
  • Подсказка про «И» и 262-ФЗ — caption2 внизу левой панели, прижата к низу.

4 · Комбинированный поиск — как устроен

Форма sud_delo принимает по одному полю на запрос, поэтому: один сетевой запрос по самому селективному заполненному полю (№ дела → УИД → ФИО), остальные дофильтровывают выдачу локально:

let primary: (SearchField, String) =
    !num.isEmpty ? (.caseNumber, num)
    : !uid.isEmpty ? (.uid, uid)
    : (.name, name)
var res = try await client.search(court:…, field: primary.0, value: primary.1)
if !num.isEmpty, primary.0 != .caseNumber { res = res.filter { $0.caseNumber == num } }
if !name.isEmpty, primary.0 != .name { res = res.filter { ($0.essence ?? "").localizedCaseInsensitiveContains(name) } }
if !uid.isEmpty, primary.0 != .uid { res = res.filter { ($0.caseUID ?? "") == uid } }

Если форма конкретного суда умеет несколько параметров сразу — фильтры можно перенести в SudrfURLBuilder, интерфейс не изменится.

5 · Отдельное окно и PDF

// открыть акт в отдельном окне (снимок, не ссылка на модель)
openWindow(value: ActWindowPayload(caseNumber: r.caseNumber, actText: model.actText))

// PDF: NSSavePanel → NSAttributedString (CourtActFormatter) → NSTextView
// → NSPrintOperation(jobDisposition: .save) — AppKit сам режет на страницы A4
ActPDFExporter.save(caseNumber: r.caseNumber, text: model.actText)
  • Payload — Codable & Hashable: повторный вызов с тем же делом поднимает уже открытое окно вместо дубля.
  • PDF — постраничный A4 (595×842 pt): поля 70/56/56/64 (слева запас под подшивку), выключка по формату с переносами, та же типографика, что в ActTextView (центрированные заголовки, курсивные глаголы, красная строка) — оба рендера питаются одним CourtActFormatter.
