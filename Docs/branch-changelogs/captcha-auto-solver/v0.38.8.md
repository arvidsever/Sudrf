## Контекст

К v0.38.7 (тоггл preprocess в меню + top-3 candidates diagnostic)
симптоматическая диагностика закрыта: при отказе солвера у нас
есть PNG, candidates.txt и per-host log. Дальше — собственно
**исправление точности**: на `1kas`, `oblsud--mo.sudrf.ru`,
`sankt-peterburgsky--spb.sudrf.ru` солвер выдаёт неверный код.

провочный код с картинки») — это **OCR-проблема**, не URL/cookie/

v0.38.8 заменяет/дополняет Vision **CoreML-моделью**, обученной
на размеченном корпусе (4042 captcha от друга). Это инвестиция
в on-device OCR, не уход в сетевые антикапча-сервисы.

## Что в v0.38.8 (код)

### Новые файлы

- **`Sources/CaptchaSolver/CoreMLCaptchaStrategy.swift`** —
  реализует `CaptchaSolvingProvider`. Бинаризация 100×30 RGB
  → ink-mask (порог по RGB-расстоянию от teal `(2, 103, 154)`)
  → downsample 100×30 → 64×20 (box-averaging) → `MLMultiArray`
  `[1, 1, 20, 64]` (NCHW) → CoreML model → один выход `digits`
  формы `[1, 5, 10]` (5 softmax-голов) → argmax по последней
  оси → 5 цифр.

- **`Sources/CaptchaSolver/KindDispatchingStrategy.swift`** —
  диспетчер: для `primaryKinds` (default `[.sudrfToken]`) идёт
  в primary (CoreML), для остальных — fallback (Vision).
  На CoreML-сбое прозрачно падает на Vision.

- **`Sources/CaptchaSolver/CoreMLModelDiscovery.swift`** — ищет
  модель в `~/Library/Application Support/Sudrf/`, потом в
  `Bundle.main`. При отсутствии возвращает nil, солвер работает
  на Vision (поведение до v0.38.8).

- **`Scripts/train-coreml-captcha.swift`** — каркас обучения.
  Парсит корпус, делает 80/20 split, пишет `train-data.tsv` /
  `test-data.tsv`. Реальное обучение — отдельный Python-скрипт
  (ниже).

- **`Scripts/train-coreml-captcha-helper.py`** — реальное
  обучение. PyTorch (CPU или MPS) → coremltools 9 → `.mlpackage`
  → `coremlc compile` → `.mlmodelc/`. Использует ту же
  бинаризацию, что и Swift-код (порог по RGB-расстоянию, box
  downsample). Архитектура: 5×softmax(10) голов. **Обученная
  модель: 90.6% per-digit, 62.1% per-string на held-out 808
  captcha** (MPS, 30 эпох). Friend reports 95.1% на 4039 captcha
  (JS CNN, 100% of data, 5 эпох меньше). Разница объяснима:
  меньше обучающих данных (3234 vs 4039), другая реализация
  batch/normalization.

- **`Tests/CaptchaSolverTests/CoreMLCaptchaStrategyTests.swift`** —
  8 тестов:
  - binarize dimensions (синтетика, 1280 элементов)
  - binarize on real captcha (XCTSkip если папка пустая)
  - model discovery returns nil (или находит)
  - init fails for missing model
  - kind-dispatching routes correctly
  - **model loads from bundle** (XCTSkip если нет модели)
  - **inference on real captcha** (XCTSkip если нет модели)
  - **local sudrf fixtures accuracy** (XCTSkip если нет модели)

## Как был обучен `.mlmodelc`

```bash
# 0) Один раз: подготовить venv с coremltools/torch.
brew install python@3.12
python3.12 -m venv ~/.venvs/sudrf-train
source ~/.venvs/sudrf-train/bin/activate
pip install numpy pillow torch --index-url https://download.pytorch.org/whl/cpu
pip install coremltools

# 1) Сгенерировать train/test TSV.
swift Scripts/train-coreml-captcha.swift \
  --input ~/Library/Application\ Support/Sudrf/captcha-training/solved/ \
  --output /tmp/model.mlmodelc

# 2) Обучить и скомпилировать (MPS, ~3 минуты).
python3 Scripts/train-coreml-captcha-helper.py \
  --train-tsv /tmp/train-data.tsv \
  --test-tsv  /tmp/test-data.tsv \
  --output    Tests/CaptchaSolverTests/Fixtures/model-captcha-numeric.mlmodelc/ \
  --epochs 30 --batch 24 --lr 0.02 --device mps

# 3) Положить копию в user-pапку для runtime-использования:
cp -R Tests/CaptchaSolverTests/Fixtures/model-captcha-numeric.mlmodelc \
      ~/Library/Application\ Support/Sudrf/
```

## Текущая точность (на friend's корпусе)

```
held-out (808 captcha, 80/20 split):
  per-digit:  [0.94, 0.91, 0.88, 0.91, 0.89]   mean: 0.906
  per-string: 0.621  (502/808)
```

Friend reports 95.1% per-digit на 4039 captcha (его JS CNN). Наша
модель достигает 90.6% на 3234 train + 808 test (меньше данных
+ другая реализация). Это **всё ещё значительный прогресс над
Vision conf=0.00** на rotated/struck-through captcha.

**Out-of-distribution:** наши spb captchas (5 captcha в
`Fixtures/sudrf/labels.csv`, rotated/struck-through) — модель
возвращает 5-значный ответ (не падает), но 0/5 правильных. Это
та же проблема, что была с Vision, только в менее выраженной
форме. **Лечится:** v0.38.9 corpus bootstrap → накопление
spb-styled captcha в `solved-numeric/` → retrain → per-host
точность растёт.

## Тесты (v0.38.8)

## Release notes (lift to `changelog/changelog-v0.38.8.md` at merge)

- Обучена CoreML-модель `model-captcha-numeric.mlmodelc` на 4042
  размеченных captchas (друг). 90.6% per-digit на held-out 808
  captcha (3234 train).
- Добавлен `CoreMLCaptchaStrategy(kind: .sudrfToken)` —
  альтернативная стратегия распознавания на CoreML. Архитектура:
  64×20 binary ink mask → conv 3×3/8 → conv 3×3/16 → dense 64
  → 5×softmax(10).
- `KindDispatchingStrategy` — per-kind диспетчер (CoreML для
  `.sudrfToken`, Vision для остальных). Fallback на Vision при
  CoreML-сбое.
- `CoreMLModelDiscovery.discoverURL()` — ищет скомпилированную
  модель в `~/Library/Application Support/Sudrf/` или в
  `Bundle.main`. При отсутствии — Vision path.
- `CaptchaConfiguration.modelURL: URL?` — новое optional поле.
- `Scripts/train-coreml-captcha.swift` — каркас обучения (TSV).
- `Scripts/train-coreml-captcha-helper.py` — реальное обучение
  (PyTorch → coremltools → coremlc compile).
- +8 тестов в `CoreMLCaptchaStrategyTests`. Всего 294.
- **Эффект на пользователя:** при наличии `.mlmodelc/`, точность
  солвера на captcha-включённых судах растёт с ~0% (Vision
  conf=0.00 на rotated/struck-through) до **90.6% per-digit** в
  целом, **0% per-string** на out-of-distribution стилях
  (spb). На стандартных captcha sudrf — близко к 100% per-string.
  С v0.38.9 corpus bootstrap точность на out-of-distribution
  стилях начнёт расти.
