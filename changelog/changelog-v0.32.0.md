# Изменения — Alpha 0.32.0

## Параллельное обновление по судам

Обход отслеживаемых дел был очень медленным: время ≈ «число дел × 1.5 с». Причина —
два последовательных горлышка: глобальный троттл в `SudrfClient` (пауза 1.5 с между
*любыми* запросами, даже к разным судам) и строго последовательный обход в
`RefreshCenter`.

Ключевой факт: в системе СОЮ у каждого суда свой отдельный сервер, поэтому троттлить
нужно только в рамках одного суда, а разные суды можно обновлять параллельно.

### Пер-хост троттл

`SudrfClient.lastRequestAt: Date?` → `nextAllowedAt: [String: Date]` (ключ — хост из
URL). `throttle(host:)` резервирует слот **атомарно до `await`**: параллельные запросы
к одному хосту честно встают в очередь с шагом 1.5 с, а к разным судам идут
одновременно. Заодно устранена скрытая гонка прежнего read-sleep-write.

`VSRFClient`/`MosGorSudClient` не тронуты — у каждого один хост.

### Параллельный обход

`RefreshCenter.refreshAll` группирует дела по домашнему суду (`displayDomain`) и запускает
один последовательный воркер на суд через `withTaskGroup`, одновременно не более
`RefreshSettings.maxConcurrentCourts = 8` судов (seed + refill по мере завершения).
Внутри суда дела по-прежнему идут по одному → пер-хост троттл держит 1.5 с.

Прогресс обхода теперь считается по завершению каждого дела (`bumpWalkProgress`,
`@MainActor`) — счётчик по индексу линейного цикла при параллелизме не годился. Логика
отмены/поколений (`walkGeneration`, `Task.isCancelled`) и защита кэша не изменились.

## Результат

Время обхода падает с «сумма всех дел × 1.5 с» примерно до времени самого нагруженного
одного суда. При этом отдельный сервер суда беспокоим не чаще прежнего (≥ 1.5 с внутри
суда), одновременно активно максимум 8 судов.

## Совместимость

Изменения только во внутренней механике обновления; формат кэша, контекста и модели не
затронуты.
