# Изменения — Alpha 0.7.0

## Симптом

Если дело прошло кассацию и было возвращено на новое рассмотрение, в
апелляционном суде по одному УИД заводятся **две** записи — исходный круг и
новый (после возврата). В движении показывался только **первый** найденный круг,
второй терялся.

Проверено ранее на реальном УИД `11RS0001-01-2025-011255-03`: ВС Коми
возвращает по нему `33-4818/2025` и `33-2266/2026`.

## Причина

`MovementService.movement` по каждому вышестоящему суду брал из выдачи только
первую строку (`results.first`) и сразу выходил (`break`). Остальные записи под
тем же УИД игнорировались.

Сопутствующая проблема: id акта строился как `act_<домен>` — для двух кругов из
одного суда он совпадал, и второй акт затирал первый (и в списке актов, и в
`actBodies`).

## Исправлено

- **`Movement.swift` — перебор всех кругов.** По каждому вышестоящему суду теперь
  обрабатываются **все** найденные по УИД записи (с непустыми id карточки), а не
  только первая. Каждый круг — отдельная инстанция и отдельный акт. Круги
  упорядочиваются по дате (старый → новый); устойчивая сортировка инстанций по
  уровню этот порядок сохраняет.
- **Уникальный id акта.** `act_<домен>` → `act_<домен>#<№ дела>`. Акты двух кругов
  больше не схлопываются. То же — в пути ручного ввода кода капчи
  (`SearchModel.ingestCaptchaCard`).
- **Классификация круг/ЧЖ — по вкладке «Обжалование» 1-й инстанции.** Частная
  жалоба живёт под тем же УИД, но кругом апелляции не является. Авторитетный
  источник различения — вкладка «Обжалование решений, определений» в карточке
  1-й инстанции (парсится из уже загруженной карточки, без доп. запросов). У
  каждой жалобы там есть явное поле **«Вид жалобы (представления)»**
  (Апелляционная / Частная / Кассационная / Замечание на протокол), вышестоящий
  суд, даты движения («Направлено в вышестоящую инстанцию», «Возвращено…»,
  «Дата рассмотрения жалобы») и результат.

  Новый парсинг: `CaseCardParser` достаёт записи вкладки в `CaseCard.appeals`
  (`[AppealRecord]`) и заодно `CaseCard.decisionDate` («Дата рассмотрения»).
  `MovementService.isRoundOfAppeal(row:card:appeals:)` сшивает запись
  вышестоящего суда с жалобой по датам (дата рассмотрения наверху = «Дата
  рассмотрения жалобы»; дата поступления наверх = «Направлено в вышестоящую
  инстанцию») и берёт её «Вид»: апелляционная/кассационная → круг, частная/прочее
  → не круг. Это надёжнее прежнего (категория — это существо спора, а апелляционный
  акт и у круга, и у ЧЖ называется «Апелляционным определением»), и работает даже
  когда наверху ещё нет результата.

  Если по датам жалоба не сшилась (нет вкладки/расхождение дат) — откат к
  различителю по «Результату рассмотрения» (`isPrivateComplaintByResult`:
  ОПРЕДЕЛЕНИЕ → ЧЖ, РЕШЕНИЕ/ПРИГОВОР → круг; при пустом результате — не отсекаем).

  Проверено на реальных карточках: горсуд (дело 2-3671/2025) даёт 5 жалоб
  [апелляц., замечание, апелляц., апелляц., **частная**]; частная (рассм.
  02.10.2025) сшивается с карточкой ВС 33-4820/2025 (дата рассм. 02.10.2025) и
  отсекается; круг 33-4818/2025 (дата рассм. 06.10.2025) сшивается с
  апелляционной жалобой и показывается.

UI правок не потребовалось: `CaseMovementView` и переключатель актов уже
перебирают `instances`/`acts` через `ForEach` по id — две апелляционные инстанции
с разными № дел рендерятся как два блока, два акта — как две строки.

- **Хронологический порядок инстанций (кассация между кругами).** Раньше блоки
  сортировались строго по уровню (1-я → апелляция → кассация), и кассация всегда
  уходила в конец. Но по делу 2-7212/2025 кассация (8Г-2430/2026, рассм.
  02.03.2026) отменила определение 1-го круга и направила на новое апелляционное
  рассмотрение — то есть по хронологии она стоит МЕЖДУ кругом 1 (оконч. 07.10.2025)
  и кругом 2 (нач. 01.04.2026). Сортировка инстанций и актов переведена на
  хронологическую: 1-я инстанция всегда первой, далее вышестоящие — по дате начала
  производства (самое раннее событие движения; для актов — по дате акта), уровень
  лишь как тай-брейк (`instanceOrderKey` / `actOrderKey`). Та же сортировка теперь
  и в пути ручного ввода капчи (`SearchModel.ingestCaptchaCard`) — иначе после
  подгрузки кассации она бы снова встала в конец.

## Тесты

`MovementServiceTests` дополнен (мок-клиент обобщён — отдаёт несколько строк
выдачи и карточки по `case_id`):

- **`testBothAppealRoundsShown`** — выдача ВС возвращает два круга (нарочно в
  порядке «новый раньше»); проверяется, что показаны оба, в хронологии
  (`33-4818/2025`, затем `33-2266/2026`), и что акты кругов имеют разные id и оба
  тела на месте.
- **`testCassationOrderedBetweenAppealRoundsByDate`** — кассация встаёт между
  кругами апелляции по датам (1-я → круг1 → кассация → круг2).
- **`testPrivateComplaintNotShownAsRound`** — выдача содержит круг и частную
  жалобу под тем же УИД; в инстанциях остаётся только круг.
- **`testObzhalovanieParsedFromRealCard`** — на реальной карточке горсуда
  (`sgs_card.html`) парсер вкладки «Обжалование» даёт 5 жалоб с верными видами и
  датами.
- **`testClassificationViaObzhalovanie`** — классификация по вкладке: частная
  жалоба (ВС 33-4820/2025) сшивается по дате и кругом не считается; апелляционная
  запись сшивается и считается кругом, даже если результат «определенческий».
- **`testRealPrivateComplaintCardClassified`** / **`testRealAppealRoundNotClassifiedAsComplaint`**
  — реальные карточки ВС: даты парсятся, фолбэк по результату работает.
- **`testAppealKindMapping`** / **`testResultFallbackHeuristic`** — юнит-проверки
  «Вид жалобы» → тип и фолбэка по результату.
- Прежняя регрессия `testHigherInstancesSearchedByCardUIDNotLinkGUID` сохранена:
  теперь круг 33-4818/2025 классифицируется как апелляция через вкладку
  «Обжалование» (а не только фолбэком).

> Сборку и `swift test` нужно прогнать у себя (в контейнере нет Swift-тулчейна).
