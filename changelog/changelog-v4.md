# Что изменилось (v4 — починка подробных карточек дел)

## Корень проблемы

`name_op=case` — это **карточка дела** (метаданные + движение + вкладки), а не
страница текста акта. Прежний `CaseCardParser` брал `body.text()` всей карточки
и резал фрагмент от первого «УИД» до «опубликовано». Но первый «УИД» — это
подпись в шапке метаданных, поэтому в «текст акта» попадала простыня из
метаданных, движения и сторон. Плюс таблица движения на этом движке —
«**событие первое**» (колонки: Наименование события | Дата | Время | …), а старый
парсер искал дату в колонке 0 и отбрасывал все строки → движение выходило пустым.

Проверено на реальных карточках (одно гражданское дело по трём инстанциям):
Сыктывкарский горсуд (1-я инст.), ВС Республики Коми (апелляция), 3-й КСОЮ
(кассация). Набор и **порядок вкладок различаются** по инстанциям, поэтому
позиционные эвристики в принципе ненадёжны.

## Как теперь

`Sources/SudrfKit/CaseCardParser.swift` переписан и разбирает карточку по
смыслу контейнеров, а не по их номерам:

| Что | Откуда берётся |
|---|---|
| УИД, судья, результат, дата поступления, категория | контейнер вкладки с меткой «Уникальный идентификатор дела» (вкладка «ДЕЛО»/«ПРОИЗВОДСТВО») |
| номер дела | заголовок карточки «ДЕЛО № …» / «ПРОИЗВОДСТВО № …» (у КСОЮ — «8Г-2430/2026 [88-4097/2026]») |
| движение (`sessions`) | таблица с заголовком `<th>` «ДВИЖЕНИЕ ДЕЛА» или «СЛУШАНИЯ»; колонки сопоставляются по названиям |
| тексты актов (`acts`) | инлайн-блоки `<div id="cont_doc{N}">` под ярлыками `<li id="tab_doc{N}">` («Судебный акт #N (тип)») |

## Изменённые файлы

| Файл | Что изменилось |
|---|---|
| `Sources/SudrfKit/CaseCardParser.swift` | Полностью переписан (см. выше). |
| `Sources/SudrfKit/Models.swift` | `CaseCard` дополнен полями `uid`, `caseNumber`, `category`, `receiptDate`, `acts: [CaseActText]`; добавлена модель `CaseActText`. Старые поля и инициализатор сохранены (обратная совместимость). `actText` теперь = текст первого акта. |
| `Sources/SudrfKit/SudrfURLBuilder.swift` | `cardURL(...)` получил параметр `new` (по умолчанию «0») — карточка апелляции/кассации без него не открывается. |
| `Sources/SudrfKit/SudrfClient.swift` | `fetchCard(...)` получил параметр `new` (по умолчанию «0») и передаёт его в `cardURL`. |
| `Sources/SudrfKit/Movement.swift` | Оба вызова `fetchCard` передают `new` из соответствующей картотеки. |
| `Sources/SudrfApp/SearchModel.swift` | Одиночная загрузка карточки передаёт `new` картотеки. |
| `Sources/sudrf-cli/SudrfCLI.swift` | У команды `card` добавлена опция `--new`. |

## Тесты

`Tests/SudrfKitTests/CaseCardParserTests.swift` — разбор трёх реальных карточек
из `Tests/SudrfKitTests/Fixtures/` (подключены как ресурсы в `Package.swift`):
проверяются УИД, номер дела, судья, результат, число строк движения и тип/начало
текста акта для 1-й инстанции, апелляции и кассации.

> Собрать и проверить: `swift test` (или ⌘U в Xcode). Логика парсера выверена на
> приложенных страницах; компилятор Swift в среде подготовки правок недоступен,
> поэтому при первой сборке возможны мелкие правки сигнатур SwiftSoup.

## Замечание по доменам (на будущее)

Питерские суды используют схему с двойным дефисом: `vos--spb.sudrf.ru`. Если в
`CourtDirectory` адреса записаны как `vos.spb.sudrf.ru`, карточки/выдача работать
не будут. Здесь не трогал — скажите, если нужно пройтись по справочнику судов.
