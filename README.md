# SudrfKit

Каркас ядра для поиска дел и судебных актов на сайтах судов общей юрисдикции
(ГАС «Правосудие», домены `*.sudrf.ru`) — нативно на Swift, без браузера.

Это перенос логики навыка `sudrf-case-search` в обычный код. Браузер навыку
нужен только из-за ограничений песочницы Claude (прокси режет `sudrf.ru`,
веб-загрузчик ломает cp1251). На вашем Mac запросы идут напрямую через
`URLSession`, поэтому достаточно прямых HTTP-запросов.

## Состав

- `SudrfKit` — библиотека:
  - `Cyrillic1251` — кодек windows-1251 и percent-кодирование query (через CoreFoundation).
  - `Court`, `CourtLevel` — суд и звено.
  - `Cartoteka`, `CartotekaRegistry` — карта параметров картотек из навыка.
  - `CaseIndexClassifier` — единый каталог индексов номеров дел (районные,
    субъектовые, апелляционные, кассационные и военные суды): отрасль,
    процессуальная роль карточки (дело / материал / жалоба) и политика связи
    материала с родительским делом.
  - `SudrfURLBuilder` — сборка URL формы / выдачи (`name_op=r`) / карточки (`name_op=case`).
  - `SudrfClient` — `actor`: запросы, троттлинг, cookies, декодирование cp1251, гейт по капче.
  - `ResultsParser`, `CaseCardParser`, `CaptchaDetector` — разбор HTML (SwiftSoup).
  - `KoAPProceduralRole` — процессуальная роль КоАП-карточки по звену суда,
    картотеке и происхождению сквозного УИД (`MS` / `RS`).
  - `CourtDirectory`, `DistrictCourtResolver`, `MagistrateDirectory` — справочники
    и живой резолвер судов (субъекты/КСОЮ/АСОЮ, районные/городские, мировые участки).
  - `MagistrateClient`, `VSRFClient`, `MosGorSudClient` — отдельные клиенты и парсеры
    для мировых участков (`*.msudrf.ru`), ВС РФ и Мосгорсуда, у которых своя вёрстка.
- `CaptchaSolver` — on-device авто-решатель капчи (Vision + CoreML, без сети);
  ядро SudrfKit от него не зависит, солвер подключает только приложение.
- `SudrfApp` — оконное SwiftUI-приложение поверх ядра (см. ниже).
- `sudrf-cli` — исполняемая утилита (подкоманды `search`, `card`, `route`,
  `district`, `harvest`).
- Тесты: `SudrfKitTests`, `SudrfAppTests`, `CaptchaSolverTests`.

## Сборка и запуск

Требуется Xcode 26+ (SDK macOS 26 «Tahoe»); сборка пакета — на macOS 26+:
интерфейс SudrfApp построен на Liquid Glass (`glassEffect`,
`.buttonStyle(.glass)`), минимальная платформа пакета — macOS 26.

CoreML-модель числовой капчи не хранится в git — она поставляется
неизменяемым Release-ассетом `model-v1` и подтягивается скриптом
(без неё модель-зависимые тесты помечаются `XCTSkip`):

```bash
bash Scripts/fetch-model.sh --asset-tag model-v1 \
  --fixtures-dir Tests/CaptchaSolverTests/Fixtures
```

```bash
swift build
swift test

# Пример из навыка: Сыктывкарский горсуд, дело об АП 5-470/2026
swift run sudrf-cli search

# Поиск по № дела в другой картотеке
swift run sudrf-cli search --type g1 --number "2-1234/2026"

# Поиск по ФИО
swift run sudrf-cli search --type u1 --name "Новожилов"

# Другой суд / звено
swift run sudrf-cli search --domain vs--komi.sudrf.ru --level subject --type g2 --number "33-100/2026"

# Карточка дела (капчей не защищена)
swift run sudrf-cli card --case-id 98765 --case-uid ABC-123-GUID --delo-id 1500001

# Маршрутизация по коду субъекта: суд субъекта + апелляционный и кассационный суды ОСЮ
swift run sudrf-cli route --subject-code 11   # 11 — Республика Коми
# Суд субъекта:  Верховный суд Республики Коми — vs.komi.sudrf.ru
# Апелляция ОСЮ: Второй апелляционный суд — 2ap.sudrf.ru
# Кассация ОСЮ:  Третий кассационный суд — 3kas.sudrf.ru
```

## Справочник судов

`CourtDirectory` — встроенный справочник, сгенерированный из справочника
КонсультантПлюс: ВС республик, краевые/областные суды, города федерального
значения, автономии (84 суда), а также 9 КСОЮ и 5 АСОЮ с территориальной
подсудностью. Для этих звеньев домен берётся из справочника, без веб-резолва:

```swift
let court = CourtDirectory.cassationCourt(forRegion: "Коми")?.court   // 3kas.sudrf.ru
let subj  = CourtDirectory.subjectCourt(matching: "Свердлов")?.court
let known = CourtDirectory.court(forDomain: "https://2ap.sudrf.ru/")
```

Нюансы: районные/городские суды в справочнике КонсультантПлюс отсутствуют —
их домены резолвятся отдельно. У части судов субъектов указан не sudrf.ru-домен
(`nnoblsud.ru`, `www.oblsud.penza.ru`, `www.uloblsud.ru`, `www.mos-gorsud.ru`) —
для них модуль `sud_delo` может быть недоступен по этому хосту (флаг
`isSudrfPlatform`); нужен sudrf-зеркальный домен.

Мировые судьи — отдельное звено `MagistrateDirectory`: список судебных участков
подтягивается вживую из справочника ГАС «Правосудие» `go_ms_search` по региону.
Участки на `*.msudrf.ru` доступны для поиска через `MagistrateClient` (номер дела
и ФИО; поиска по УИД в форме участка нет), а московские, петербургские и другие
внешние порталы остаются в списке помеченными как неподключённые.

## Принципиальные ограничения

- **Капча решается человеком по умолчанию, с опциональным авто-солвером.**
  При обнаружении капчи клиент бросает `SudrfError.captchaRequired(formURL:)`.
  Без солвера сценарий: открыть форму в браузере, ввести код вручную
  (через `CaptchaAssistSheet`). Карточка дела (`name_op=case`) капчей
  не защищена — имея `case_id`/`case_uid`, акты тянутся свободно.
  Дополнительно поставляется отдельный продукт `CaptchaSolver`
  (Vision framework, без сетевых вызовов). По умолчанию **включён**;
  отключается переключателем в системном меню «Captcha» (`⌃⌘A`).
  При низкой уверенности или N неудачных попытках клиент возвращается
  к ручному вводу. Распознавание — локальное и опциональное: проект
  не делает это обязательным, и ядро SudrfKit не зависит от солвера.
  Точность на простых цифровых капчах sudrf (spb, мск) близка к 100%
  по `VisionOCRStrategy`; на зашумлённых/повёрнутых картинках (nsk и
  некоторые винтажные суды) чаще падает в ручной режим.
- **Селекторы выдачи best-effort.** `case_id`/`case_uid` извлекаются из href
  надёжно; привязка остальных ячеек строки сделана по позиции и может
  потребовать подстройки под текущую вёрстку конкретного суда (тест работает на
  фикстуре). Текст акта берётся от метки «УИД …» до конца карточки.
- **Официального API де-факто нет** — это парсинг, контракт неустойчив, разметку
  и параметры площадка может менять; ядро рассчитано на сопровождение.
- **262-ФЗ.** Часть категорий не публикуется: пустой результат ≠ отсутствие дела.
- **Уважительный доступ.** Включён троттлинг (`minInterval`), задан User-Agent —
  поправьте контакт в `SudrfClient.init`.

## Что проверено

Байтовые векторы cp1251 сверены с навыком: «Найти» → `%CD%E0%E9%F2%E8`,
«а» → `%E0`, «2а-3021/2023» → `2%E0-3021%2F2023`. Они закреплены в тестах
`Cyrillic1251Tests`. Сетевой слой на «живых» судах не прогонялся — это делается
на вашей машине.

## Приложение (SwiftUI)

Поверх ядра есть оконное приложение `SudrfApp` (таргет в этом же пакете).
Каркас — `NavigationSplitView` со стеклянным сайдбаром из четырёх разделов
(`AppSection`):

- **Обзор** — сводка по отслеживаемым делам, ближайшие заседания,
  процессуальные сроки и лента изменений (`OverviewView`).
- **Мои дела** — отслеживаемые дела с группировкой (списком / по стадиям /
  по производствам / по подборкам), хранилище `TrackedStore`, фоновое
  обновление и уведомления (`RefreshCenter`, `FeedNotifier`).
- **Поиск** — поиск дела с пикером суда: регион → суд из резолвера (включая
  мировые участки) → картотека → поле/запрос → список результатов → карточка
  с текстом акта; двойной клик по строке разворачивает **движение дела** по
  инстанциям, собранное по УИД (`Movement*`).
- **Календарь** — заседания отслеживаемых дел по датам, режимы
  «Месяц · Неделя · Повестка» с дорожкой сроков (`CalendarScreen`).

Дополнительно:

- **Импорт из CSV** (`CaseImport`) — перенос дел из выгрузки стороннего
  сервиса (LegalHelp, см. `Scripts/export_cases_csv.py`): карточки разных
  инстанций и материалов группируются по сквозному УИД и сшиваются в одно
  дело.
- **Починка цепочек** (`TrackedCaseRepairCoordinator`, `CaseOriginResolver`) —
  идемпотентное восстановление сохранённых дел: слияние дублей, поиск
  нижестоящих карточек по официальной ссылке «Рассмотрение в нижестоящем
  суде», точному номеру и сквозному УИД. Полная цепочка «первая инстанция →
  материал → апелляция → кассация» сохраняется целиком; автоматическая связь
  создаётся только при единственном точном совпадении.
- **CAPTCHA** — сначала пробует on-device авто-солвер (`CaptchaSolver`,
  Vision + CoreML; включён по умолчанию, настройки — в системном меню
  «Captcha», ⌃⌘A). При неудаче или низкой уверенности открывается нативный
  лист **CAPTCHA Assist**: крупная картинка и поле кода, полная страница суда
  живёт в скрытом `WKWebView`. Ожидающие ввода карточки группируются по хосту
  суда — один введённый код разблокирует всю группу. Кроме обычных судов ОСЮ
  поддержаны отдельные клиенты для мировых участков (`*.msudrf.ru`), ВС РФ и
  Мосгорсуда, у которых своя вёрстка.

Запуск (от лучшего к простому):
1. **Xcode-проект (рекомендуется)**: `brew install xcodegen && xcodegen`,
   затем `open Sudrf.xcodeproj`, схема **Sudrf**, ⌘R — настоящий .app-бандл
   с Liquid Glass (конфигурация — в `project.yml`). Без XcodeGen: в Xcode
   File → New → Project → macOS App «Sudrf», удалить шаблонные файлы,
   File → Add Package Dependencies → Add Local… → эта папка (продукт SudrfKit),
   и добавить файлы из `Sources/SudrfApp` в таргет.
2. **Скрипт**: `bash Scripts/make-app.sh` — соберёт и откроет
   `build/SudrfApp.app` без Xcode-проекта.
3. `swift run SudrfApp` — быстрый запуск без бандла: macOS 26 рендерит
   такой бинарь в режиме совместимости (без части стекла).

## Версионирование

Пока приложение в альфе, публичная версия пишется как `Alpha 0.N.P`.
Текущая версия — `Alpha 0.40.0` (build 80): второе число — номер
фича-релиза внутри ветки `0.x`, третье — патч без новой функциональности.
Заметки по версиям — в `changelog/changelog-v*.md`, релизы с готовыми
сборками — на странице
[Releases](https://github.com/arvidsever/Sudrf/releases).

- Новая пользовательская фича или заметное расширение поведения:
  `0.37.0` → `0.38.0`.
- Только исправления багов: `0.37.0` → `0.37.1`, затем `0.37.2`.
- Релиз с фичей и багфиксами получает feature bump, а в changelog разделяется
  на «Добавлено», «Изменено», «Исправлено» и «Тесты».
- Стабильный публичный релиз станет `1.0.0`; `Beta`/`RC`/`Stable` — это метка
  качества рядом с версией, а не отдельный сегмент номера.

В app bundle `CFBundleShortVersionString` хранит публичную версию
(`0.37.1`), а `CFBundleVersion` — внутренний монотонный билд (`38`, затем
`39`, `40` и дальше). Changelog релизов называется в формате
`changelog/changelog-v0.37.1.md`, архивы сборок — вроде
`build/Sudrf-Alpha-0.37.1-build38.zip`.

При каждой новой версии нужно обновить `MARKETING_VERSION` и
`CURRENT_PROJECT_VERSION` в `project.yml`, затем заново сгенерировать
Xcode-проект командой `xcodegen`. Если релиз собирается через
`Scripts/make-app.sh`, значения `RELEASE_CHANNEL`, `MARKETING_VERSION` и
`CURRENT_PROJECT_VERSION` в скрипте должны совпадать с `project.yml`.

## Передать сборку другому человеку

1. `bash Scripts/make-app.sh` — соберёт универсальный (Apple Silicon + Intel)
   `build/SudrfApp.app` и версионированный архив в `build/` для пересылки.
2. У получателя должна быть **macOS 26+** (минимальная версия пакета).
3. Подпись ad-hoc (без Developer ID), поэтому Gatekeeper при первом запуске
   скажет «не удаётся проверить разработчика». Получателю нужно либо
   выполнить в Терминале `xattr -cr путь/к/SudrfApp.app` и открыть, либо
   ПКМ → «Открыть», а затем Системные настройки → Конфиденциальность
   и безопасность → «Всё равно открыть».
4. Без предупреждений Gatekeeper — только подпись Developer ID + нотаризация
   (платный аккаунт Apple Developer): в Xcode — Product → Archive →
   Distribute App → Direct Distribution.

Требуется macOS 26+ (Liquid Glass). Каркас окна — NavigationSplitView со
стеклянным сайдбаром и системным инспектором (`.inspector`); кнопки —
`.glass` / `.glassProminent`. Логика — на проверенном ядре; сетевые запросы
идут напрямую с вашего Mac.

## Районные/городские суды (живой резолвер)

`DistrictCourtResolver` запрашивает у единого портала
`sudrf.ru/index.php?id=300&court_subj=<код субъекта>` список судов выбранного
субъекта, разбирает блок результатов и кэширует — в памяти и на диск, постепенно
накапливая справочник по мере запросов регионов.

```swift
let resolver = DistrictCourtResolver()
let komi = try await resolver.courts(forRegion: "Коми")          // RS — районные/городские
let mil  = try await resolver.militaryCourts(forRegion: "Коми")  // GV/OV — военные, отдельно
let all  = try await resolver.allCourts(forRegion: "Коми")
try await resolver.refresh(forRegion: "Коми")                    // перечитать субъект
```

```bash
swift run sudrf-cli district --subject-code 11              # 11 — Республика Коми
swift run sudrf-cli district --subject-code 66 --refresh    # 66 — Свердловская область
swift run sudrf-cli district --subject-code 78 --debug      # 78 — Санкт-Петербург; диагностика парсинга
swift run sudrf-cli harvest --type GV  # все гарнизонные суды одним запросом (включая зарубежные)
swift run sudrf-cli harvest --type OV  # окружные (флотские) — строки для хардкода в CourtDirectory
```

По каждому суду возвращаются домен, название и **классификационный код**
(`11RS0001`): первые 2 цифры — субъект, буквы — тип (`RS` районный/городской,
`GV`/`OV` военный, `OS` суд субъекта). Фильтрация и привязка к региону идут по
этому коду, а не по названиям. Числовой код субъекта для запроса берётся из
встроенной таблицы `subjectCodeTable` сопоставлением региона по корням слов.
Кэш по умолчанию в `Caches/SudrfKit-districts.json`.

## Куда расширять

- Картотеки — `CartotekaRegistry`: четыре набора по звеньям (`district` /
  `subject` / `appealSOYu` / `cassationSOYu`), у каждой картотеки — индексы
  номенклатуры (`prefixes`: «2а», «33», «8Г»…). `matches(caseNumber:level:)`
  подбирает картотеку по номеру дела; пары `delo_id`/`new` выверены по
  универсальному JS-переключателю видов производства из фикстур.
- Суды субъектов/АСОЮ/КСОЮ — `CourtDirectory`; районные/городские —
  `DistrictCourtResolver`; мировые участки — `MagistrateDirectory`.
- Поверх ядра — SwiftUI-интерфейс (`SudrfApp`): разделы Обзор / Мои дела /
  Поиск / Календарь, отслеживание дел, движение по инстанциям и CAPTCHA Assist.
