## Контекст

После v0.39.10 (CoreML-модель + corpus bootstrap на rebased ветке)
пользователь запустил приложение и нашёл серьёзный баг: на
captcha-включённых судах (spb, oblsud-mo, 1kas) модель **даёт
неверный код** с высокой уверенностью (conf 0.83 на spb captcha).
Токен сохраняется в `CaptchaTokenStore` и используется повторно.
Суд возвращает страницу результатов с маркером «неверный код» —
`SearchPageClassifier` корректно классифицирует её как
`.captchaRejected` (v0.39.10) — но `SudrfClient.runVariants`
**не инвалидирует токен** на этом пути, и каждый следующий search
получает ту же ошибку. Цикл до перезапуска приложения.

Дополнительно: дампы v0.39.6 использовали префикс `variant_`
для `.unrecognized` ответов, а `.captchaRejected` шёл в тот же
`variant_` — было сложно фильтровать. v0.39.12 вводит
отдельный префикс `rejected_`.

## Что в v0.39.12 (код)

### Bug fix: wrong-token feedback loop

- **`Sources/SudrfKit/SudrfClient.swift`**:
  - В `runVariants`, в `case .captchaRejected:` теперь
    `await captchaStore.invalidate(domain: court.domain)`
    (если был токен). Токен удаляется из `CaptchaTokenStore`.
    Следующий search с тем же хостом не найдёт токена, пойдёт
    обычным путём: `fetchForm` → `CaptchaDetector.hasCaptcha`
    → `captchaRequired` → manual captcha (или auto-solver попробует
    снова). Без этого фикса — бесконечный цикл с одинаковым
    плохим токеном.
  - В `runVariants` добавлен флаг `lastWasCaptchaRejected`.
    Если дамп нужен, в нижнем `if let lastData` теперь вызывается
    либо `dumpCaptchaRejected` (для отказа суда), либо `dumpVariant`
    (для неизвестного формата). Префикс `rejected_` отличает эти
    два диагноза при ручном разборе.
  - Добавлен internal `init(session:...)` для тестов —
    позволяет подсунуть `URLSession` с `URLProtocol` stub'ом
    (глобально зарегистрированные protocol classes не
    подхватываются `URLSessionConfiguration.default`).

- **`Sources/SudrfKit/MagistrateClient.swift`** (по запросу
  пользователя, тот же фикс для magistrate-пути):
  - В `search` после `fetchHTML` (до `MagistratePageClassifier`)
    проверяем `SearchPageClassifier.classify(html) == .captchaRejected`.
    При совпадении: `dumpCaptchaRejected` (тот же новый helper)
    и `throw SudrfError.searchModuleUnavailable`. Дополнительной
    инвалидации не нужно — `MagistrateURLBuilder.searchURL` НЕ
    включает captcha-параметр в URL, значит captcha в этой ветке
    не передаётся. Дамп — для диагностики.

### Diagnostic prefix

- **`Sources/SudrfKit/SearchDiagnostics.swift`**:
  - Добавлен `dumpCaptchaRejected(data:host:)` и `dumpCaptchaRejected(html:host:)` —
    префикс `rejected_` в имени файла, отличается от
    `variant_` (неизвестный формат) и `form` (предпроверка
    формы). FIFO 50 на всю папку, как и раньше.

### Tests

- **`Tests/SudrfKitTests/SudrfClientCaptchaTests.swift`** (новый, 3 теста):
  - `testCaptchaRejectedInvalidatesToken` — главный тест фикса.
    `URLProtocol` stub отдаёт страницу с маркером «неверный код».
    После `client.search` токен в `CaptchaTokenStore` отсутствует.
  - `testCaptchaRejectedDumpsWithRejectedPrefix` — дамп пишется
    с префиксом `rejected_`, не `variant_`.
  - `testCaptchaRejectedWithoutTokenDoesNotCrash` — если токен
    не был сохранён (первый search без captcha), `.captchaRejected`
    всё равно корректно пробрасывается без падения. Страховка
    от regression в `if captcha != nil` guard.

- **`Tests/SudrfAppTests/AutoCaptchaSolverTests.swift`** (фикс pollution):
  - `setUpWithError` / `tearDownWithError` подменяют
    `CaptchaSolverLog.shared` на tmp-dir логгер (паттерн
    `CaptchaSolverLogTests`). Раньше single test
    `testReturnsNilWhenDisabled` писал 3 ERROR строки про
    `example.test` в `~/Library/Application Support/Sudrf/captcha-solve.log`
    (нет DNS → 3 попытки → 3 строки). После прогона
    `swift test` production лог получал 235+ строк тестового
    шума. Проверено: до фикса = 2790 байт за один test run,
    после = 0 байт.

## Release notes (lift to `changelog/changelog-v0.39.12.md` at merge)

- **Bug fix:** `SudrfClient.runVariants` теперь инвалидирует
  captcha-токен при получении `.captchaRejected` (страница
  «неверный код»). Без этого фикса токен переиспользовался в
  бесконечном цикле — пользователь не мог выйти из этого
  состояния без перезапуска приложения. Это объясняет
  поведение, наблюдавшееся на spb/oblsud-mo/1kas после v0.39.10.
- **Diagnostic prefix:** `dumpCaptchaRejected` пишет в
  `rejected_<host>_<ts>.html` (отдельно от `variant_` для
  неизвестного формата). Ручной разбор дампов теперь явно
  отличает «суд отверг токен» от «суд вернул неизвестный формат».
- **MagistrateClient:** тот же `SearchPageClassifier.captchaRejected`
  check + `dumpCaptchaRejected` для диагностики. Captcha-инвалидация
  для magistrate не нужна (URL не содержит captcha-параметра).
- **Internal `init(session:...)` на `SudrfClient`** — для
  тестирования с `URLProtocol` stub'ом. Не используется
  в продакшене.
- **Test infrastructure:** `AutoCaptchaSolverTests` подменяет
  `CaptchaSolverLog.shared` на tmp-dir в `setUpWithError` —
  `swift test` больше не засоряет
  `~/Library/Application Support/Sudrf/captcha-solve.log`
  строками про `example.test`.
- +3 теста в `SudrfClientCaptchaTests`. Всего 310 (297 ранее +
  13 новых из main: CalendarWeekLayout, CourtDirectory, SearchPageClassifier).
- **Эффект на пользователя:** captcha-включённые суды (spb,
  oblsud-mo, 1kas) больше не зацикливаются на плохом токене.
  После одного отказа пользователь увидит manual captcha sheet
  (или auto-solver попробует снова с новым токеном). Регрессия
  против v0.39.0–v0.39.10.
