# Дорожная карта Swift 6 и AI для Sudrf

Канонический план поэтапного внедрения Swift 6, системных возможностей Apple
и генеративного AI. Документ используется как общий источник истины для
отдельных feature-веток и обновляется после merge соответствующих PR.

## Правила ведения

- Стабильные идентификаторы этапов: `S6-0`, `DATA-1`, `SPOT-2`, `INTENT-3`,
  `AI-4`, `SUMMARY-5`, `APPLE-6`, `CASEAI-7`, `PUBLIC-8`.
- Статусы: `planned`, `in_progress`, `completed`, `blocked`.
- Этап получает `completed` только после merge PR и фиксации результатов
  проверки в этом документе.
- До передачи PR коллеге автор реализации публикует в PR собственный AI
  self-review: риски, проверенные инварианты, результаты тестов и известные
  ограничения. Merge разрешён только после последующего независимого
  adversarial code review коллеги, устранения всех принятых замечаний и
  повторного прогона затронутых проверок. Результаты обоих review и
  перепроверки фиксируются в branch draft и PR.
- Каждая feature-ветка ссылается на идентификатор этапа в своём draft-файле
  `Docs/branch-changelogs/<branch>/vX.Y.Z.md`.
- Release-changelog и номера версий меняются только при merge/release.

### Обязательный merge-gate каждого этапа

1. Реализация и автоматические проверки критерия готовности.
2. AI self-review автора публикуется в открытом PR раньше review коллеги. Он
   содержит минимум: diff-scope, инварианты, риск потери данных, concurrency и
   privacy, негативные сценарии, результаты тестов и известные ограничения.
3. Независимый adversarial code review коллеги с акцентом на потерю данных, privacy,
   concurrency, ошибочные AI-выводы и деградацию fallback-путей.
4. Разбор замечаний: исправление либо явно записанное обоснование отклонения.
5. Повторный прогон всех затронутых проверок.
6. Merge PR; только после merge статус меняется на `completed`.

## Зафиксированные продуктовые решения

- Deployment target остаётся macOS 26.0; macOS 27 — активная beta/RC-линия
  разработки. API macOS 27 закрываются availability-проверками.
- Сначала выпускается личный режим с BYOK; ключи хранятся в macOS Keychain.
- Облачная обработка включается разовым явным согласием и может быть отозвана.
- В облако отправляется только акт, для которого пользователь запросил сводку;
  фоновой отправки базы нет.
- Первая генеративная функция — структурированная сводка одного акта с
  проверяемыми ссылками на оригинальные абзацы.
- Apple через английский — явно помеченный Experimental-режим, выключенный по
  умолчанию.

## `S6-0` — Swift 6 и macOS 26/27

1. Поднять tools version до Swift 6, включить complete checking и затем Swift 6
   language mode в SwiftPM и XcodeGen; фактический flip фиксировать результатами
   сборки, а не одним статусом в плане.
2. Проверять complete concurrency для всех продуктов и тестов. Для SwiftSoup,
   закреплённого на `2.7.7` из-за бага оптимизатора Swift 6.4 beta, отдельно
   отслеживать upstream и применять `@preconcurrency import` только там, где
   граница действительно нужна.
3. Сохранить изоляцию: UI/SwiftData/настройки — `@MainActor`; сеть, OCR,
   индексация, перевод и AI — actor; межакторные значения — `Sendable`.
4. Поддерживать обязательную стабильную сборку Xcode 26/macOS 26 и сборку
   новых API Xcode 27/macOS 27. Пока hosted runner не содержит Xcode 27,
   стабильная линия блокирует merge, Xcode 27 проверяется вручную, а
   opportunistic CI-job явно сообщает `not executed` и автоматически начинает
   сборку при появлении SDK — без тихой подмены Xcode 26.
5. После macOS 27 RC сделать Xcode 27 основной средой, не повышая deployment
   target до миграции тестеров.

Критерий: package, тесты и схема Sudrf собираются в Swift 6 без concurrency
warnings.

## `DATA-1` — единый слой сущностей

1. До изменения схемы ввести `VersionedSchema` и явный migration plan, сделать
   проверяемый бэкап файлов persistent store (`store`, `-wal`, `-shm`) и при
   ошибке показать блокирующий экран восстановления. Тихая рабочая сессия в
   in-memory fallback запрещена.
2. Оставить `TrackedCaseRecord` источником отслеживания, `movementData` —
   источником движения.
3. Добавить перестраиваемую SwiftData-проекцию `CourtActRecord`. `sourceActID`
   считается нестабильной подсказкой, а не гарантией портала. Логический ID
   назначается версионированным алгоритмом и сохраняется: exact source match →
   уникальный semantic match (дело/инстанция/суд/вид/дата) → уникальный
   `sourceHash` → новый deterministic ID. Исправление текста меняет
   `sourceHash` и инвалидирует сводку, но по возможности сохраняет document ID.
4. Вынести общий `ModelContainer`; добавить `CaseCatalog` actor, возвращающий
   только `Sendable`-снимки.
5. `ActParagraphizer` становится единственной границей сегментации и поглощает
   соответствующую логику `CourtActFormatter`; экран, PDF, Spotlight и AI не
   имеют независимого повторного разбиения.
6. Хранить `paragraphizerVersion` и snapshot границ/текстов абзацев вместе с
   актом и сводкой. Обновление алгоритма не перенумеровывает старые цитаты;
   новый snapshot строится только для новой ревизии `sourceHash`.
7. Перестраивать проекции после импорта, refresh, ремонта цепочки и удаления.

Критерий: существующая база мигрирует без потери данных, а ID, хеши и номера
абзацев стабильны между запусками.

## `SPOT-2` — Core Spotlight

1. Создать отделённые от SwiftData `CaseEntity` и `CourtActEntity`, реализующие
   `IndexedEntity`.
2. Индексировать реквизиты дела, стороны, категорию, подборки, события,
   результаты и полный текст актов.
3. Реализовать actor `SpotlightIndexer`: insert/update/delete/rebuild.
4. Добавить deep links на дело и выбранный акт.
5. Сначала задействовать системный Spotlight, затем глобальный поиск Sudrf через
   `CSUserQuery`, сохранив нынешний быстрый фильтр списка.

Критерий: поиск по реквизитам или смыслу открывает правильные дело и акт.

## `INTENT-3` — App Intents и onscreen awareness

Последовательность интентов: открыть дело, открыть акт, обновить дело, добавить
в подборку, показать заседания, экспортировать акт в PDF.

1. Реализовать `CaseEntityQuery` и `CourtActEntityQuery` через `CaseCatalog`.
2. Добавить русские App Shortcuts с английским fallback.
3. Зарегистрировать URL-схему и обрабатывать deep links/холодный запуск через
   существующий `@MainActor AppRouter`; отдельный `NavigationCoordinator` без
   обоснованного рефакторинга не вводить.
4. Связывать открытые дело и акт с `NSUserActivity` и AppEntity.
5. Русскую печатную Siri AI тестировать экспериментально, не делая её
   обязательным интерфейсом.

Критерий: интенты работают через Shortcuts, корректно обрабатывают CAPTCHA и
сетевые ошибки, а система получает контекст текущей сущности.

Для `RefreshCaseIntent`: при успешном автосолве дождаться обновления; если
CAPTCHA остаётся, вернуть сохранённый кэш и явный результат «требуется открыть
Sudrf», продолжив сценарий только в foreground. Интент не пытается показать
CAPTCHA внутри Shortcuts.

## `AI-4` — универсальная AI-инфраструктура

Базовый интерфейс:

```swift
protocol ActSummarizing: Sendable {
    func summarize(
        document: ActDocument,
        options: SummaryOptions
    ) async throws -> ActSummary
}
```

1. `ActSummary` хранит требования, позиции сторон, обстоятельства, мотивировку,
   резолютивную часть, суммы, даты, сроки, обжалование, предупреждения и ссылки
   на исходные абзацы.
2. Реализовать `MockActSummarizer` и BYOK/benchmark-адаптеры Groq, GigaChat и
   YandexGPT; основного провайдера не назначать заранее — его выбирает benchmark
   с учётом качества и географической доступности. Apple подключается тем же
   протоколом.
   Каталог [`nejib1/Free-LLM`](https://github.com/nejib1/Free-LLM) используется
   только для discovery кандидатов: сам
   README прямо предупреждает, что тарифы и лимиты меняются еженедельно. Перед
   добавлением каждого адаптера обязательна перепроверка по официальным docs:
   фиксируемый model ID, structured JSON/schema, контекст, доступность из РФ и
   региона разработчика, privacy/retention и условия free tier. Community
   routers и провайдеры без понятного оператора/политики обработки судебных
   актов не допускаются. OpenRouter free-router не используется. Google AI
   Studio, Cerebras и Mistral можно включить в benchmark опубликованных актов только
   после этого gate; локальные Ollama/LM Studio из списка рассматриваются как
   прототипы, а production local-путь для macOS остаётся MLX/Core ML.
3. Первый личный прогон фиксирует Groq BYOK и `openai/gpt-oss-120b`: Settings →
   AI содержит Keychain, проверку соединения, облачное согласие и подготовку
   Apple через английский. Выбор провайдера и автоматический режим возвращаются
   только после сравнительного benchmark.
4. Фиксировать model IDs; не использовать случайный free-router.
5. Benchmark на 50–100 уже опубликованных и обезличенных судом актов хранится
   вне Git; в Git остаются runner и синтетические фикстуры. Повторное
   обезличивание не выполняется: проверяется реальный опубликованный текст.
   Полный benchmark вручную, не в CI.
6. Согласие на внешний cloud уже в личном BYOK-режиме явно предупреждает об
   отправке ФИО и иных персональных данных третьих лиц.

Порог допуска: 100% существующих paragraph-ссылок, не менее 95% правильных
критических реквизитов и 90% полноты обязательных разделов.

## `SUMMARY-5` — сводка одного акта

1. Добавить действие к выбранному акту и в отдельное окно.
2. Короткий акт обрабатывать одним typed/structured запросом; длинный —
   иерархически по границам абзацев. Chunk budget является свойством модели;
   Apple-путь проектируется под малое контекстное окно и никогда не получает
   типичный длинный акт целиком.
3. Локально валидировать ссылки, дословные evidence quotes, суммы, даты, номера
   и JSON-схему; разрешить один повтор.
4. Ссылки переходят к оригинальному абзацу с подсветкой.
5. `ActSummaryRecord` хранит provider/model/promptVersion/pipelineVersion,
   `sourceHash` и `generatedAt`; при смене текста сводка устаревает.
6. Всегда показывать предупреждение о проверке по оригиналу.

## `APPLE-6` — Apple Foundation Models и двойной перевод

Режимы: Apple с русским напрямую; Apple через английский (Experimental);
внешний BYOK.

Прямой путь используется только при доступности модели и официальной поддержке
русского locale.

Переводной путь сначала имеет статус spike с go/no-go по отдельному
мини-benchmark. Полный production pipeline ниже реализуется только при `go`.

Spike проверяет загрузку языковой пары, сохранность paragraph/literal IDs,
качество ru→en→summary→ru и реальную вместимость chunk pipeline на нескольких
коротких и длинных актах.

Production-путь после `go`:

1. Проверить и при необходимости загрузить пару русский ↔ английский через
   Translation framework.
2. Извлечь из русского оригинала защищённые литералы: номера дел/УИД, даты,
   сроки, суммы, валюты и нормы права; назначить им стабильные IDs.
3. Перевести абзацы batch-запросом, сохраняя paragraph IDs вне переводимого
   текста.
4. Получить строгую английскую `ActSummary` от Apple.
5. Перевести обратно только человекочитаемые поля; JSON-ключи, IDs и реквизиты
   через перевод не пропускать.
6. Подставить оригинальные русские литералы, привязать выводы к русским
   абзацам и выполнить локальную проверку.
7. Показывать индикатор двойного перевода и диагностическую английскую сводку.
8. Кэш учитывать OS build, языковую пару и версию pipeline.

До сравнительного benchmark рабочий пользовательский маршрут зафиксирован на
Groq BYOK / `openai/gpt-oss-120b`. Apple напрямую и через английский существуют
как изолированные экспериментальные адаптеры и не подменяют рабочий провайдер.
После benchmark планируемый автовыбор: Apple напрямую → Apple через английский,
если эксперимент включён → победивший BYOK → понятное сообщение о недоступности.

## `CASEAI-7` — AI уровня всего дела

Статус: `planned`; этап на продуктовой паузе до завершения `APPLE-6`.

1. Детерминированное «Что изменилось» между двумя актами.
2. История по инстанциям на основе движения и проверенных сводок.
3. «Спросить по делу»: локально выбрать релевантные абзацы и передать только их.
4. Digest: детерминированные события, AI только поверх проверенных фактов.
5. Natural-language command bar выполняет действия через App Intents.
6. На macOS 27 использовать `SpotlightSearchTool` с fallback для macOS 26.

## `PUBLIC-8` — публичное распространение

Статус: `planned`; этап на продуктовой паузе до завершения `APPLE-6`.

1. Сохранить BYOK как пользовательский вариант.
2. Продуктовый облачный режим провести через backend proxy с авторизацией,
   квотами, бюджетами и аудитом без хранения текстов.
3. Провести privacy- и юридическую проверку актов с персональными данными.
4. Собирать только технические метрики без текста и реквизитов дела.
5. Apple через английский сохраняет Experimental до успешного benchmark после
   macOS 27 RC.

## Сквозные проверки

- Миграция SwiftData и стабильность ID/хешей/абзацев.
- Spotlight lifecycle и deep links.
- App Intents, холодный запуск, CAPTCHA и сетевые ошибки.
- Keychain, отзыв согласия и отсутствие секретов в логах.
- Structured output, chunking, retry, отмена и инвалидирование кэша.
- Translation round trip для сумм, дат, статей, имён и номеров дел.
- Сохранность paragraph IDs и literal IDs.
- Сетевой тест: провайдер получает только выбранный акт.
- Матрица Apple Silicon/Intel, macOS 26/27, Apple Intelligence и языковых
  пакетов. На Intel отсутствие Apple Intelligence и деградация к BYOK являются
  ожидаемым результатом. Translation/Apple Intelligence и полный benchmark —
  ручные проверки, а не обязательные jobs CI.
