# Изменения — v33

## Системные уведомления macOS + счётчик на иконке дока

Раньше о новом движении по отслеживаемому делу можно было узнать только открыв
приложение и заглянув в «Ленту изменений» на «Обзоре». Теперь фоновый мониторинг
(`RefreshCenter`, каждые ~10 мин) при появлении новых записей в ленте показывает нативное
уведомление macOS, а на иконке приложения в доке отображается счётчик дел с обновлениями.

## Уведомления

Новый `FeedNotifier` (`Sources/SudrfApp/FeedNotifier.swift`) — тонкая обёртка над
`UserNotifications`:

- `configure()` из `AppDelegate.applicationDidFinishLaunching` ставит делегата и запрашивает
  разрешение (`.alert`, `.sound`, `.badge`). Перед обращением к
  `UNUserNotificationCenter.current()` проверяется наличие бандла
  (`Bundle.main.bundleIdentifier != nil`) — голый SwiftPM-бинарник иначе падает.
- По одному уведомлению на новую запись ленты: заголовок — клиент (или «Дело №…»),
  подзаголовок — номер, тело — текст события. `identifier` = стабильный `id` записи, поэтому
  повторная подача того же события не создаёт дубль. Пачка ограничена (первые 10), чтобы
  большой обход не завалил Центр уведомлений.
- `willPresent` возвращает `[.banner, .sound]` — баннер виден, даже когда приложение
  активно (мониторинг часто открыт на переднем плане).
- Клик по уведомлению (`didReceive`) через колбэк `onOpen` поднимает окно и открывает
  карточку дела (`AppRouter.openCase(key:)`).

## Детект новых записей

Новая запись движения всегда получает **новый `id`** (в него входят дата/время/текст),
поэтому «новизна» определяется diff'ом против персистентного множества уже виденных id
(`knownFeedIDs`, `UserDefaults` ключ `notifiedFeedIDs.v1` — по образцу `readFeedIDs`).

`AppRouter.reload()` получил параметр `notifyNew: Bool = false`. Значение по умолчанию
сохраняет все существующие вызовы: добавление/импорт дел лишь тихо поглощают новые id, без
уведомлений. Уведомляет только `applyRefreshed` (результат **фонового** обновления),
вызывая `reload(notifyNew: true)`. Для открытого сейчас дела `RefreshCenter` не сбрасывает
`seenAt`, значит его записи остаются прочитанными и не порождают дубль-уведомление.

## Бейдж дока

`FeedNotifier.setBadge` выставляет `NSApp.dockTile.badgeLabel` в число дел с обновлениями
(`AppRouter.newBadge`, дела с `seenAt == nil`) — тот же сигнал, что рисует чип «обновлено»
в списке. Бейдж пересчитывается на каждом `reload()`: открытие дела (`markSeen` → `reload`)
уменьшает его, а при старте он восстанавливается.

## Совместимость

Локальные уведомления на macOS не требуют спец-entitlement или ключей Info.plist; App
Sandbox в проекте не включён. Формат кэша, контекста и моделей не затронут.
