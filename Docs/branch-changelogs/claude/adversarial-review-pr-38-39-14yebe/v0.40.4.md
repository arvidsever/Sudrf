## Исправлено

- `testConcurrentRequestsReserveSeparateThrottleSlots` переведён с проверки
  интервалов между `startLoading` URLProtocol на проверку суммарного времени
  трёх конкурентных запросов монотонными часами (`ContinuousClock`).
  Ослабленные в v0.40.3 пороги (`minInterval / 3`) всё равно упали на CI:
  лаг доставки одного запроса достигает полного `minInterval`, и соседний
  наблюдаемый интервал сжимается до ~0.0003с при исправном троттле (main,
  run 98) — любой положительный порог на отдельный интервал флакует.
- Новый инвариант детерминирован в сторону фейла: резервация слотов до
  `await` гарантирует, что третий запрос спит до `T0 + 2×minInterval`, а
  `Task.sleep` не просыпается раньше срока — elapsed ≥ 2×minInterval×0.9.
  При регрессии «резервация после await» все запросы уходят сразу и elapsed
  составляет единицы миллисекунд. Лаги CI только увеличивают elapsed и
  ложный фейл невозможен.
- Продакшн-код `VSRFClient` по-прежнему не менялся.
