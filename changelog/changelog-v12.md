## v12.8 — зарубежные ГВС: подведомственность подтверждена экспертом

Пять гарнизонных судов «Территорий за пределами РФ» переведены из

## v12.7 — кодовая карта гарнизонных (выгрузка GV) + дефисные домены повсюду

**Гарнизонные — теперь по полным классификационным кодам** (выгрузка
эксперта, court_type=GV, 104 суда: 99 по РФ + 5 зарубежных). Кодовая карта
`garrisonOkrugDomainByCode` (99 записей) — первичный ключ lookup'а; имя
(466-ФЗ) — второй; региональный префикс — последний фолбэк. Полный код —
единственный честный ключ: префикс врёт не только у 224 ГВС (39GV0005 →
1-й Западный), но и у Ярославского (77GV0013, «московский» → 2-й Западный).
Сверка выгрузки с картой 466-ФЗ нашла и потерю: Ярославский ГВС выпал при
ручной перепечатке списка в генератор v12.6 — возвращён, полнота закреплена
тестом на 99. Зарубежные ГВС (109/26/40/5/80, «Территории за пределами РФ»,
в 466-ФЗ отсутствуют) идут региональным фолбэком по коду приписки
(66GV0008 «109 ГВС» → Центральный) — корректность приписки желательно
подтвердить экспертом.

**Дефисные домены — универсально.** Синонимия host.region ↔ host--region
действует на всей платформе (sankt-peterburgsky--spb и т.п.), и sud_delo
работает именно на дефисном варианте. Новый хелпер
`CourtDirectory.dashVariant(of:)`; все сетевые запросы приложения (поиск,
карточка, движение) идут через дефисный домен (`CourtOption.searchCourt`),
отображается «точечный»; вышестоящим в движении передаются оба варианта.
Точечный спецкейс «vs.» из v12 удалён за ненадобностью.


## v12.6 — подсудность гарнизонных по 466-ФЗ; справочник субъектов сверен с порталом

**Исправлена ошибка v12.5** (спасибо эксперту): маршрут «гарнизонный → ОВС»
строился по коду субъекта из классификационного кода, но код отражает
дислокацию-историю, а НЕ юрисдикцию. Контрпример: 224 ГВС — код 39
(«калининградский»), а юрисдикция по закону — часть Санкт-Петербурга и
Ленинградской области → вышестоящий 1-й ЗОВС, не Балтийский флотский.

**Карта по каждому суду из закона**: ст. 1 ФЗ от 29.12.2020 № 466-ФЗ
(ред. 03.04.2023) — все 98 гарнизонных военных судов разобраны по субъектам
юрисдикции и сведены к ОВС через 345-ФЗ. Проверено: ни одна юрисдикция не
пересекает границу округа ОВС, маппинг однозначен. Ключ —
нормализованное НАЗВАНИЕ суда (`garrisonKey`): падежеустойчиво
(«224-го гарнизонного военного суда» из закона = «224 гарнизонный военный
суд» с портала), регион в скобках из портальной выдачи отрезается, цифры
сохраняются. Региональный код — только фолбэк для имён вне карты.
Регресс-тесты: 224-й → 1-й Западный; Курильский → 1-й Восточный (не
Тихоокеанский флотский!); Воркутинский → 1-й Западный; полнота карты (98)
и принадлежность всех доменов списку ОВС.

**Справочник судов субъектов сверен с живой выгрузкой портала** (webarchive
от эксперта, court_type=OS, 89 судов):
- добавлены отсутствовавшие: Севастопольский горсуд (92, gs.sev),
  Запорожский облсуд (90, oblsud.zpr), ВС ДНР (93, vs.dnr),
  ВС ЛНР (94, vs.lnr), Херсонский облсуд (96, oblsud.hrs);
- Пензенский (58) и Ульяновский (73) областные переехали на платформу
  sudrf (oblsud.pnz / oblsud.uln) — старые внеплатформенные домены
  из справочника КонсультантПлюс заменены;
- остальные 82 кода сверены — расхождений нет (Коми: портал даёт

## v12.5 — данные из первоисточников вместо догадок (портал + КонсультантПлюс)

Портал и КонсультантПлюс оказались доступны напрямую — все гадания заменены
снятыми фактами:

**Окружные (флотские) военные суды — их ДЕВЯТЬ, не шесть** (живая выдача
`court_type=OV&court_subj=0`): шесть окружных + три флотских (Балтийский,
Северный, Тихоокеанский). Все домены и классификационные коды — с портала;
шаблонные шесть доменов v12.4, к слову, совпали, но флотских в модели не было.

**АВС/КВС — живые домены**: vap.sudrf.ru и vkas.sudrf.ru (шаблонные догадки
avs.mo/kvs.nsk были неверны). `sud_delo` на vap проверен — работает.
Эвристика «домен → звено» в движении дела знает оба.

**Подсудность из законов, по региональным кодам:**
- ОВС — ст. 1 ФЗ от 27.12.2009 № 345-ФЗ (ред. 27.11.2023): полная карта
  «код субъекта → ОВС», все 89 кодов накрыты (регресс-тест). Ключевое:
  закон распределяет юрисдикцию по субъекту ДИСЛОКАЦИИ, а классификационный
  код гарнизонного суда несёт именно его (224 ГВС в СПб — 39GV0005 →
  Балтийский флотский) — поэтому **военная вертикаль движения дела теперь
  строится**: гарнизонный → его ОВС → КВС; окружной (1-я инстанция) → АВС → КВС.
- КСОЮ — ст. 23.1 ФКЗ-1 (ред. 7-ФКЗ от 31.07.2023): новые субъекты
  (90 Запорожская, 93 ДНР, 94 ЛНР, 96 Херсонская) — **Второй КСОЮ**;
  остальная сгенерированная карта подтверждена.
- АСОЮ — ст. 23.9 ФКЗ-1: новые субъекты — **Первый АСОЮ**; остальное
  подтверждено.

Не добыто (фетчер залип на кэше портала): домены Севастопольского горсуда
и судов новых субъектов — добываются у вас одной командой
`sudrf-cli harvest --type OS`, строки готовы для вставки в
`subjectCourts` + `subjectCourtDomainByCode`.


## v12.4 — упрощение по схеме эксперта: типовые запросы + хардкод подсудности

Подсказка пользователя: портал отдаёт ВСЕ суды типа одним запросом —
`…&court_subj=0&court_type=GV…`. Общероссийский 91-субъектный скан удалён.

**Живой резолв остаётся ровно двум спискам:**
- районные/городские — по региону (как раньше);
- гарнизонные — один типовой запрос `court_type=GV` (`garrisonCourts()`),
  включая зарубежные (они не относятся ни к одному субъекту). Открытие
  военной ступени теперь стоит один HTTP-запрос вместо 91.

**Все вышестоящие — хардкод** (`CourtDirectory`): 6 окружных (флотских)
военных судов, Апелляционный и Кассационный военные суды. Живьём подтверждён
домен 1-го Западного (78OV0000, проба по СПб); остальные домены — по шаблону
именования военных сайтов ГАС, проверяются одной командой
`sudrf-cli harvest --type OV` (и AV/KV/OS) — она печатает готовые строки
DirectoryCourt для замены. Той же командой добываются недостающие суды
субъектов: Севастополь (92) и новые регионы (90/93/94/96) в исходном
справочнике отсутствуют.

**Подсудность — по региональным кодам, не словами.** Захардкожены таблицы:
- код субъекта → номер КСОЮ (`cassationCourt(forSubjectCode:)`);
- код субъекта → номер АСОЮ (`appealCourt(forSubjectCode:)`);
- код субъекта → домен суда субъекта (`subjectCourt(forSubjectCode:)`)
  + обратная (`subjectCode(forDomain:)`).
Сгенерированы из территориальной подсудности справочника, коллизии разведены
вручную («22» ВС Алтая ↔ Алтайский краевой; «50» облсуд ↔ «77» Мосгорсуд).
Код принимается и в полном виде («78RS0015» → «78»). Движение дела подбирает
вышестоящие ТОЛЬКО по этим таблицам: районный суд — из своего
классификационного кода, суд субъекта — обратной таблицей по домену.
Словесное корневое сопоставление АСОЮ/КСОЮ (v12.3) удалено вместе с
Мосгорсуд-хаком. Регресс-тест: каждый код с судом субъекта обязан иметь
КСОЮ и АСОЮ. Новые субъекты в картах подсудности не расписаны — дополняются
по мере уточнения (см. комментарии в коде).

Карта «гарнизонный суд → его окружной» сознательно не закладывалась:
военная подсудность экстерриториальна, сквозное движение военных дел
не строится.


## v12.3 — регион только у районных; военные суды единым списком (+ зарубежные)

**Пикер региона остаётся только у районных/городских судов** — все остальные
ступени выдают полный список сразу:

- Областные и приравненные — все 80+ из справочника (вне-sudrf-платформенные
  помечены); АСОЮ — 5; КСОЮ — 9. Прежний выбор сохраняется, список из одного
  суда выбирается сам, иначе «— выберите —».
- **Военные ступени — общероссийский скан портала**: один проход по ВСЕМ
  записям портальной таблицы субъектов, включая «Территории за пределами РФ»
  (код 95 — зарубежные гарнизонные суды, которые не относятся ни к одному
  субъекту и раньше были недостижимы). Первый скан при штатном троттлинге
  (1,5 с/запрос) занимает ~2–3 минуты, прогресс виден в статусной строке
  («Собираю военные суды: N/91 субъектов…»); результат кэшируется на диск —
  дальше мгновенно. Сбой отдельного субъекта скан не валит: субъект
  перечитается при следующем обращении. Гарнизонные/окружные/АВС/КВС —
  фильтры по типу кода (GV/OV) и названию (АВС/КВС) поверх единого списка.
- CLI: `sudrf-cli military` — тот же скан с разбивкой по типам (удобно
  прогреть кэш заранее и проверить буквы кодов АВС/КВС вживую).

**Движение дела от суда субъекта**: регион в пикере погашен, поэтому
АСОЮ/КСОЮ подбираются по НАЗВАНИЮ выбранного суда — новое корневое
сопоставление `appealCourt/cassationCourt(matchingRoots:)` («Свердловский
областной суд» → Свердловская область → Седьмой КСОЮ). Прежний
`regionsMatch` подстрочный и названия судов с регионами не сопоставлял.
Спецслучай: Мосгорсуд через корень «москов» цеплял Московскую ОБЛАСТЬ
(Первый КСОЮ) вместо города Москвы (Второй) — ключ нормализуется.

## v12.1 — «не подсасываются районные суды» (репорт по Санкт-Петербургу)

Разбор показал ДВА пласта проблем; СПб-кейс добивается диагностикой на месте.

**1. Сопоставление регион → портальный код (ломало пять регионов, не СПб).**
Полный прогон всех имён пикера через матчер выявил: Курская, Томская области
и Республика Марий Эл давали ПУСТЫЕ корни слов («Курская» → «кур» съедался
фильтром длины) — молчаливый ноль судов; Республика Алтай маппилась на
Алтайский край, Сахалинская область — на Республику Саха (ничьи скоринга,
выигрывал первый по таблице). Починено: (а) точное имя таблицы — мгновенный
ответ без эвристик (пикер передаёт ровно их, чинит все 91); (б) окончание
стрижётся, только если остаётся корень ≥ 4 букв; (в) точное совпадение корня
весит 3, префиксное пересечение — 1 («сахалин» ≠ «саха»). Регресс-тест гоняет
ВСЕ регионы таблицы на самосопоставление.

**2. Хрупкости пайплайна резолвера (любая даёт ровно «молча пусто»).**
- Пустой/сбойный ответ портала навсегда помечал субъект «загруженным» —
  регион залипал до ручной чистки кэша. Теперь пустой ответ не помечает,
  а «отравленный» старым кэшем субъект перечитывается сам; файл кэша — v2.
- Фильтр судов шёл по первым двум цифрам классификационного кода: код иного
  формата/отсутствует → суд исчезал. Теперь каждый суд несёт метку
  `portalSubject` (каким court_subj добыт) — фильтр по ней, код лишь фолбэк
  для старых кэшей.
- Код нераспознанного формата (`CourtKind.other`) выпадал из выдачи районных;
  теперь, если RS-судов нет, показываются суды с нераспознанным кодом.

**3. Диагностика для добивания СПб (портал из песочницы недоступен).**
`swift run sudrf-cli district --region "Город Санкт-Петербург" --debug` —
печатает URL запроса, размер HTML, число якорей `a.court-result`, сколько
записей отброшено и почему (нет кода / нет sudrf-домена), распределение по
типам, встреченные коды, намёк на пагинацию (ровно 20 якорей) и расхождение
«код ≠ портальный субъект». Если суды в отчёте есть — в приложении они
появятся (фильтрация починена); если якорей ноль — портал сменил разметку,
присылайте HTML.


В сайдбаре поиска появились переключатели **вида судов** (Общие / Военные,
сегмент-контрол) и **звена** — по пять ступеней на ветвь, сверху вниз:

| Общие | Военные |
|---|---|
| Верховный Суд РФ *(задел)* | Верховный Суд РФ *(задел)* |
| Кассационные суды ОЮ | Кассационный военный суд |
| Апелляционные суды ОЮ | Апелляционный военный суд |

Картотеки в пикере теперь зависят от звена (наборы v11 наконец доступны из
интерфейса, а не только районный). При смене звена невалидная картотека
сбрасывается на первую доступную; авто-выбор картотеки по индексу № дела
работает на уровне текущего звена (в КСОЮ «8Г-…» переключит на гражданскую
кассацию и т.д.).

## Откуда берутся суды каждой ступени

- **Районные/городские** — живой резолвер портала (как раньше, коды RS).
- **Областные и приравненные** — встроенный справочник: у региона ровно один
  суд, он подставляется сам; для судов вне платформы sudrf (Мосгорсуд,
  Нижегородский, Пензенский, Ульяновский областные) в названии — пометка.
- **АСОЮ / КСОЮ** — все 5/9 из справочника, преселектится суд, обслуживающий
  выбранный регион (по территориальной подсудности).
- **Гарнизонные** — резолвер, фильтр по буквенному типу кода **GV**.
- **Окружные (флотские)** — фильтр **OV**; портал перечисляет их в регионе
  размещения штаба, поэтому в большинстве регионов список пуст — статус
  подсказывает выбрать регион штаба.
- **АВС / КВС** — по одному на страну: резолвятся фильтрами **AV**/**KV** в
  регионах размещения (Московская область / Новосибирск), пикер региона
  для них выключен.
- **ВС РФ** — заглушка: список судов пуст, кнопка «Искать» неактивна, статус
  объясняет, что у ВС РФ отдельный портал (vsrf.ru) и парсинг не подключён.

## Ядро (SudrfKit)

- `Court.swift`: новые `CourtBranch` (general/military) и `CourtTier`
  (5 ступеней, `level` → `CourtLevel?`, у ВС РФ — nil; `title(branch:)`).
  Военные суды работают на той же платформе и с той же номенклатурой,
  поэтому наборы картотек берутся по соответствующему `CourtLevel`.
- `DistrictCourt.codeLetters` — буквенный тип классификационного кода
  (11RS0001 → «RS»); `DistrictCourtResolver.militaryCourts(forRegion:codes:)`
  — фильтр по типам GV/OV/AV/KV.
- `CourtDirectory.subjectCourt(forRegion:)` — суд субъекта по региону через
  машинерию `regionCode` (корни слов + алиасы Москвы/Петербурга).
- `MovementService.courtLevel(forDomain:)`: домены АСОЮ вида `Nap.sudrf.ru`
  теперь распознаются как `.appeal` (раньше падали в `.subject`).

## Движение дела — домены вышестоящих больше не захардкожены

`SearchModel` строит `MovementService` под каждый «провал» в дело:
для районных дел общих судов — суд субъекта региона (оба варианта домена,
`vs.X` и `vs--X`; мёртвый сервис молча пропускает) + КСОЮ региона; для дел
суда субъекта — АСОЮ + КСОЮ региона. Обобщение, обещанное в v5, сделано.
Для военных судов сквозное движение пока не строится: доменов окружных
судов и АВС/КВС нет во встроенном справочнике (они резолвятся живьём).

   с кодами AV/KV (фильтры написаны по структуре кода, живьём не гонялись).
2. Поиск по картотекам военных судов (платформа та же, но live-прогона не было).
3. Двойной вариант домена субъекта в движении дела (vs.X / vs--X).
