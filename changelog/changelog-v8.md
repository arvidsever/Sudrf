# Что изменилось (v8 — АП: подтягиваются ВС и второй круг после возврата)

## Симптом

Дело об АП с возвратом на новое рассмотрение показывалось только первым кругом.
Реальный пример (УИД `11RS0001-01-2023-002356-90`, Сыктывкарский горсуд):

1. **12‑255/2023** — горсуд (судья Печинина), картотека «Жалобы по делам об АП»
   (`delo_id=1502001`, `adm1_case`): жалоба на определение/решение прокуроров
   удовлетворена, дело возвращено в прокуратуру.
2. **21‑183/2023** — ВС Республики Коми (судья Пешкин), картотека «АП: жалобы на
   решения по жалобам» (`delo_id=1513001`, `adm2_case`): по протесту прокурора
   решение судьи горсуда отменено, дело направлено на новое рассмотрение.
3. **12‑544/2023** — горсуд (судья Леконцев), та же `adm1_case`: прокурорские
   акты оставлены без изменения.

В карточке движения был виден только круг 1. ВС не подтягивался, второй круг
горсуда — тоже. Вместо ВС в блоке вышестоящих висела заглушка «Третий
кассационный суд» (форма под капчей), хотя дело в КСОЮ не ходило.

## Причина

Две независимые недоработки:

1. **ВС не опрашивался.** `MovementService.higherCartotekaIDs(...)` на уровне суда
   субъекта возвращал картотеки только для `g/u/p`; для базовой картотеки
   `admj`/`adm` — пустой список, поэтому суд субъекта по делу об АП вообще не
   опрашивался. Вдобавок самой картотеки ВС для АП — «жалобы на решения по
   жалобам» (`1513001`/`adm2_case`) — не было в `CartotekaRegistry.subject`.
   А заглушка «Третий кассационный суд» — побочный эффект: для АП на уровне
   кассации маппинг `adm→adm33` есть, 3‑й КСОЮ опрашивался, упирался в капчу и
   оставлял заглушку.

2. **Второй круг домашнего суда не искался.** `movement(...)` строил первую
   инстанцию из одной открытой карточки и искал по УИД **только вышестоящие**
   суды. Повторное рассмотрение в том же суде после возврата (новая карточка
   `adm1_case` под тем же УИД) никогда не искалось.

## Исправлено

- **Картотека ВС для АП.** В `CartotekaRegistry.subject` добавлена `adm2`
  «АП: жалобы на решения по жалобам» (`delo_id=1513001`, `new=0`,
  `delo_table=adm2_case`, поля `adm2_case__CASE_NUMBERSS` /
  `adm2_case__JUDICIAL_UIDSS` / `adm2_parts__NAMESS`). Значения — из навыка
  `sudrf-case-search`, сверены с реальной карточкой ВС (дело 21‑183/2023).

- **Маппинг картотек.** `higherCartotekaIDs(baseID:level:)` на уровне субъекта/
  апелляции теперь отдаёт `["adm2"]` для базовых `adm`/`admj` (протест/жалоба на
  не вступившее решение по делу об АП → суд субъекта). За вступившие в силу акты
  по‑прежнему отвечает КСОЮ (`adm33`).

- **Переопрос домашнего суда по УИД (все круги первой инстанции).**
  `movement(...)` после базовой карточки ищет по УИД в том же суде и той же
  картотеке и добавляет каждую найденную карточку с **другим** № дела отдельной
  инстанцией 1‑й инстанции. Базовый круг исключается сравнением номеров с учётом
  дописок выдачи («… ~ М‑…») — `sameCaseNumber(_:_:)`. Правка **общая** (не только
  АП): покрывает и возврат гражданского/уголовного/КАС‑дела кассацией в тот же
  суд на новое рассмотрение.

- **Строго хронологический порядок инстанций и актов.** Снято жёсткое закрепление
  1‑й инстанции сверху: `instanceOrderKey`/`actOrderKey` сортируют **только** по
  дате (начало производства / дата акта), уровень — тай‑брейк. Иначе при возврате
  (горсуд → ВС → горсуд) оба круга горсуда вставали бы до ВС. Теперь порядок —
  горсуд‑круг1 → ВС → горсуд‑круг2. Прежняя хронология «кассация между кругами
  апелляции» сохраняется (там 1‑я инстанция и так самая ранняя).

- **Заголовки актов по АП.** `actTitle` различает: жалоба на постановление по
  делу об АП в райсуде (`admj`) → «Решение» (а не «Постановление», как было); АП
  во 2‑й инстанции суда субъекта (`adm2`) → «Решение» (а не «Апелляционное
  определение»). Постановление по делу об АП по существу (`adm`) — по‑прежнему
  «Постановление».

UI не правился: `CaseMovementView` рендерит `instances` через `ForEach`, две
карточки 1‑й инстанции с разными № дел и судьями показываются как два блока.

> Заглушка «Третий кассационный суд» по делам об АП остаётся (КСОЮ под капчей):
> это валидная возможность вручную проверить пересмотр вступившего в силу акта,
> а не ошибка. При необходимости подавить её для АП — отдельной правкой.

## Тесты (`MovementServiceTests`)

Мок обобщён: различает домашний суд (по домену) и вышестоящие — по УИД домашний
отдаёт «другие круги той же инстанции», вышестоящие — свои записи; ожидаемый УИД
параметризован.

- **`testKoAPRemandRoundsHomeCourtAndSubject`** — полная КоАП‑цепочка
  (12‑255 → 21‑183 (ВС) → 12‑544): подтянуты оба круга горсуда и ВС; порядок
  строго хронологический; базовый круг не продублирован; акты с разными id и
  телами, в хронологии дат.
- **`testKoAPSubjectCartotekaRegistered`** — `adm2` в реестре с верными
  платформенными значениями.
- **`testKoAPHigherCartotekaMapping`** — `admj/adm → adm2` на уровне субъекта;
  гражданский маппинг (`g1 → g2`) не сломан.
- **`testKoAPActTitles`** — заголовки актов АП.
- Существующая регрессия `testHigherInstancesSearchedByCardUIDNotLinkGUID`
  ослаблена по числу запросов (теперь по УИД опрашивается и домашний суд), но
  по‑прежнему проверяет, что в поиск уходит УИД, а GUID ссылки — никогда.

> Сборку и `swift test` прогоните у себя (в контейнере нет Swift‑тулчейна).
