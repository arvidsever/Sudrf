# Изменения — v28

## Воронеж: ATS-исключение и primary-фолбэк; Благовещенск: карточка по ссылке выдачи

Живые тесты v26 (скриншоты + два webarchive пользователя): Ульяновск заработал;
Воронеж падал уже с «-1200 Не удалось установить безопасное подключение из-за
ошибки TLS»; Благовещенск искал с капчей, но карточка была пустой, а движение —
минимальным.

## 1. Воронеж

Webarchive Коминтерновского районного суда показал два факта. Во-первых,
Safari открывает суд по https — значит, -1200 в приложении даёт **App Transport
Security** (URLSession приложения обязан требовать TLS ≥1.2 + forward secrecy,
Safari этим правилам не подчиняется; у воронежского сервера слабый TLS —
рукопожатие рвётся ещё ДО делегата, поэтому мягкая проверка сертификата из v26
не помогала). Во-вторых, выдача открыта **современным (primary) паттерном** —
классификация Воронежа как «винтажного» в срезе апстрима устарела.

- `project.yml` → Info.plist: `NSAppTransportSecurity` с исключениями для
  `sudrf.ru` (TLS от 1.0, без forward secrecy, разрешён insecure HTTP — задел
  под http-фолбэк, сейчас НЕ используется) и `mos-gorsud.ru` (без forward
  secrecy). CLI и тесты ATS не ограничивает — только приложение.
- `SudrfURLBuilder.rawSearchURLVariants`: перебор вариантов винтажного суда
  теперь замыкается primary-вариантом — если суд на самом деле работает на
  современном модуле, перебор сам его найдёт, а `WorkingVariantStore` запомнит.

## 2. Благовещенск: cardURL-first

Webarchive карточки показал: винтажная выдача этого суда даёт ссылку на
карточку ТОЛЬКО с `_uid` — без `_id`
(`name_op=case&_uid=…&_deloId=…&_caseType=0&_new=0&srv_num=1&_hideJudge=0`).
Наши проверки требовали ОБА идентификатора → «У записи нет идентификаторов
карточки» и minimalMovement. При этом сама ссылка выдачи самодостаточна.

- `SudrfClient.fetchCard(url:)` — карточка по готовой ссылке из выдачи (без
  host-фолбэка: URL пришёл с уже отвечавшего хоста). Метод добавлен в протокол
  `CaseProviding` (моки тестов дополнены).
- `MovementService`: хелперы `hasCardAccess(row:)` («есть пара ID или ссылка»)
  и `fetchCard(row:court:cartoteka:)` (пара ID → канонический путь через
  билдер, переживающий смену формы хоста; иначе — ссылка выдачи). Применены во
  всех местах, где карточка грузится по строке выдачи: базовая карточка, круги
  того же суда, материалы, вышестоящие инстанции, кассация московской ветки.
  Прямые ссылки импорта (KnownCard) — прежний путь.
- `SearchModel.openCard`: без пары ID карточка открывается по `cardURL`.

## 3. Парсер винтажной карточки: второй живой образец

Благовещенская карточка вскрыла вариативность винтажной разметки:
- настольная таблица вкладки помечена то «non-list» (Ульяновск), то «list»
  (Благовещенск) — надёжный маркер только `none-mobile` (мобильный дубль в
  `div.block-mobile` его лишён); селектор заменён;
- колонки движения называются то «Дата события»/«Время события», то
  «Дата»/«Время слушания» — маппинг шапки расширен (+ «Место проведения» → зал);
- ключ судьи — просто «Судья» (уже поддерживалось), категория с `<br>` и
  стрелками, стороны семейного спора «Информация скрыта».

Актов в обеих живых карточках нет — распознавание винтажной вкладки актов
по-прежнему TODO (нужен webarchive карточки с опубликованным актом; блоки
`doctab_content_*` видны в JS-шаблоне страницы).
