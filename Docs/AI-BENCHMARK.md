# Локальный benchmark сводок судебных актов

Корпус 50–100 уже опубликованных судами актов, ключи и ответы провайдеров
хранятся **вне Git**. Полный
benchmark запускается вручную: CI не имеет доступа к корпусу, облачным ключам,
Apple Intelligence и установленным Translation language packs.

## Подготовка одного акта

1. Сохранить опубликованный судом текст в локальный каталог вне репозитория.
2. Не выполнять повторное автоматическое обезличивание: benchmark должен
   проверять тот же текст, который реально увидит пользователь Sudrf.
3. Зафиксировать локально URL публикации, hash исходника, категорию/длину акта
   и эталонные
   критические суммы, даты, сроки и результат.

## Provider gate

Перед прогоном сверить по официальной документации: точный model ID, лимит
контекста, structured output, географию, retention/privacy и актуальность free
tier. `nejib1/Free-LLM` служит только каталогом кандидатов. Router/community
endpoint без известного оператора и политики обработки не используется.

## Порог

- 100% ссылок указывают на существующие ¶ID, evidence quote дословен;
- ≥95% критических сумм, дат, сроков и результатов верны;
- ≥90% обязательных разделов заполнены, когда факт есть в оригинале.

Результат измеряется отдельно для коротких и длинных актов, каждого provider /
model ID, macOS build и (для Apple spike) языкового пути.

`SummaryBenchmarkRunner` читает JSON-массив фикстур, запускает тот же
`ConfiguredActSummarizer`, что используется приложением, и считает три порога
выше. Формат показан в `Docs/AI-Benchmark-Fixtures/synthetic.json`; настоящий
корпус следует хранить по тому же формату вне Git. Синтетическая фикстура нужна
для теста runner, а не для оценки юридического качества модели.
